# Go Guard — Non-Negotiable Rules

These rules apply to every Go file written, reviewed, or generated by this engine.
They are not guidelines. They are not suggestions. They are invariants.

## Error Handling

- NEVER ignore errors with `_` (the only exception: `defer f.Close()` when the
  error is truly irrelevant AND documented with a comment)
- ALWAYS wrap errors with context: `fmt.Errorf("doing X: %w", err)`
- NEVER log AND return the same error
- NEVER use `panic` for control flow outside of application startup

## Concurrency

- NEVER start a goroutine without a known exit path
- ALWAYS pass `context.Context` as the first parameter to I/O functions
- ALWAYS run `go test -race` before declaring code complete
- NEVER use `time.Sleep` for synchronization in production code

## Style

- ALWAYS use `goimports` formatting (which includes `gofmt`)
- ALWAYS organize imports in three groups: stdlib, external, internal
- NEVER use dot imports (`. "pkg"`)
- ALWAYS use composite literal field names (never positional)
- NEVER exceed 4 function parameters — use an options struct

## Architecture

- NEVER put business logic in HTTP handlers
- ALWAYS inject dependencies through constructors
- NEVER use package-level mutable state (`var db *sql.DB`)
- ALWAYS define interfaces at the consumer, not the provider

## Testing

- ALWAYS write table-driven tests for multi-case scenarios
- ALWAYS use `t.Run` for subtests
- NEVER test private functions directly — test through public API
- ALWAYS include both success and error path test cases

## Security

- NEVER use `math/rand` for security-sensitive operations
- NEVER build SQL queries with string concatenation
- NEVER hardcode secrets in source code
- ALWAYS use parameterized queries (`$1`, `$2`)
