# Shell and Terminal

## Purpose

This bridge skill routes developer-tool work into the canonical shell-and-terminal doctrine.

Use it when the task is about:

- shell scripting posture
- terminal environment strategy
- portability and path safety
- tmux or session-oriented workflows
- evidence capture and resumable shell flows
- runtime-shell boundaries for shell behavior

## Reading Path

1. Read `../../SKILL.md` for the engine boundary.
2. Read `../../shell-and-terminal/README.md`.
3. Read `../../shell-and-terminal/INVENTORY.md` and `../../shell-and-terminal/TRIGGER_SCOPE.md` if maturity or trigger ownership is unclear.
4. Read `../../references/shell-and-terminal/INDEX.md`.
5. Read `../../references/shell-and-terminal/shell-terminal.md`.
6. Read `../../references/shell-terminal-mastery.md` and `../../references/shell-and-terminal/runtime-shell-governance.md` when environment strategy or runtime boundaries matter.
7. If the task has mutation or destructive risk, also read `../../rules/shell-safety/non-destructive-defaults.md`.

## Pressure Classification

Classify the shell pressure first:

- scripting law
- terminal posture
- portability target
- session continuity or evidence capture
- runtime-shell boundary
- safety enforcement

## Output Contract

Return:

1. primary shell lane
2. why it belongs here first
3. strongest truth surface (doctrine, command, or rule)
4. one anti-pattern to avoid
5. next exact doctrine file or runtime surface to use

## Core Rule

Keep shell work explicit, portable, and path-safe.
This bridge skill exists to keep shell-facing work routed through doctrine and safety law before automation convenience takes over.
