# Language Specialists

## Purpose

This bridge skill routes developer-tool work into the canonical language-specialists doctrine.

Use it when the task is about:

- language-native idioms outside dedicated heavy engines
- specialist ecosystem truth that still belongs inside developer-tool
- deciding whether the real issue is language-local, platform-shaped, interop-shaped, or should escalate into another engine
- preserving native language truth without turning developer-tool into fake ecosystem completeness

## Reading Path

1. Read `../../SKILL.md` for the engine boundary.
2. Read `../../language-specialists/README.md`.
3. Read `../../language-specialists/INVENTORY.md` and `../../language-specialists/TRIGGER_SCOPE.md` if maturity or first-owner routing is unclear.
4. Read `../../language-specialists/ABSORPTION_MATRIX.md` if donor promotion or routing placement is part of the problem.
5. Read `../../references/language-specialists/INDEX.md`.
6. Read `../../references/language-specialists/language-specialists.md`.
7. If the problem is really platform or interop shaped, route outward to `../../references/cross-platform-development/INDEX.md` or the owning engine.

## Pressure Classification

Classify the primary specialist pressure first:

- native idiom or style
- framework or toolchain reality
- platform boundary
- interop boundary
- heavy-engine escalation

## Output Contract

Return:

1. primary language-specialist lane
2. why it belongs here first
3. whether the task should stay bridge-shaped or escalate outward
4. one anti-pattern to avoid
5. next exact doctrine file or runtime surface to use

## Core Rule

Bridge first, imitate never.
This bridge skill exists to preserve ecosystem-specific truth while routing deeper platform, interop, or engine-core pressure to the correct broader doctrine owner.
