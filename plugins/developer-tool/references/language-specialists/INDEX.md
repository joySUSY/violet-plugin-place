## Reading Paths

### If you need the bridge model first

1. `language-specialists.md`
2. then branch by the dominant specialist pressure

### If the task is truly language-idiom specific but not heavy-engine sized

1. `language-specialists.md`
2. preserve the language-native idiom and toolchain truth
3. escalate only if the task becomes runtime-, interop-, or engine-core-heavy

### If the task is really a platform or support-tier problem

1. `language-specialists.md`
2. `../cross-platform-development/INDEX.md`
3. `../platform-infrastructure.md` if the issue becomes more about platform organization than coding idiom

### If the task is really an interop or runtime-boundary problem

1. `language-specialists.md`
2. route outward to the owning lane instead of deepening the specialist note itself
3. use `../../agents/language-specialists-diagnostician.md` only when bounded diagnosis adds leverage

### If the task has outgrown bridge guidance and belongs to a heavy engine

1. `language-specialists.md`
2. route to `../../rust-coding-engine/SKILL.md` or `../../typescript-coding-engine/SKILL.md`
3. continue in that engine instead of forcing `developer-tool` to pretend it owns the deeper ecosystem truth

### If the task is mobile/native-shell adjacent

1. `language-specialists.md`
2. `../cross-platform-development/INDEX.md`
3. then choose whether the next owner is platform doctrine, interop doctrine, or a dedicated heavy engine

## Language-Specialists Decision Questions

Before choosing a subpage, ask:

1. Is the real pressure about **native language idiom**, **framework/runtime trade-offs**, **platform boundaries**, or **heavy-engine escalation**?
2. Is this still a bridge-level question inside `developer-tool`, or already a Rust-, TypeScript-, cross-platform-, or interop-owned problem disguised as a language question?
3. Do we need the bridge doctrine first, or are we already certain the task belongs elsewhere?
4. Are we preserving native language truth, or trying to route into deeper system doctrine?

The doctrine is:
- language-specialists docs are organized by specialist pressure and routing pressure
- not by whichever language keyword appeared first
