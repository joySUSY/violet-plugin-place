# Developer Tool References Index

## Purpose

Canonical navigation entrypoint for the `developer-tool/references` doctrine tree.

This index exists so the `developer-tool` engine can be read by doctrinal responsibility rather than by historical file sprawl or donor memory.

Use this index first when the task is about:

- plugin or runtime shell architecture
- agentic workflow systems and orchestration
- memory or continuity surfaces
- shell and terminal behavior
- build, release, deployment, or supply-chain posture
- deterministic data processing and structured output pipelines
- cross-platform strategy and compatibility trade-offs
- bridge-shaped language specialist guidance

This page exists to answer a routing question:

> given a real developer-tool task, which doctrinal lane should be loaded first, which supporting lanes usually follow, and how do the engine's orchestration, runtime-shell, memory, shell, build, data, platform, and specialist surfaces fit together as one system?

## Root Navigation Questions

Before diving into a subpage, ask:

1. Is this primarily an **orchestration** question, a **memory** question, a **runtime-shell** question, a **shell/terminal** question, a **delivery** question, a **data-processing** question, or a **platform/specialist** question?
2. Is the real pressure about **activation**, **runtime behavior**, **continuity**, **surface selection**, **release truth**, **pipeline stages**, or **support-tier fidelity**?
3. Do I need the root lane doctrine first, or am I already certain which specialized page owns the problem?
4. Is this problem still local to `developer-tool`, or should it escalate into another heavy engine or runtime surface?

The doctrine is:
- the root index routes by problem pressure
- not by whichever filename or donor memory looks familiar first

## Cross-Lane Rules

### Orchestration before automation
If the problem is about agent behavior or runtime activation, start in `agentic-system-basis/` before designing commands or hooks.

### Memory before shell gimmicks
If the problem is about continuity or recall, start in `memory-systems-overview.md` / `ai-agent-memory/` before reaching for runtime tricks.

### Tool-ecosystem before component sprawl
If the task is about plugin surfaces, start in `tool-ecosystem/` before deciding “this should be a command/agent/hook/MCP”.

### Shell before clever commands
If the task is shell-facing, start in `shell-and-terminal/` before optimizing for convenience.

### Build/deploy before ritualized CI
If the task is delivery-facing, start in `build-and-deploy/` before discussing specific pipeline syntax or release scripts.

### Data-processing before orchestration theater
If the task is about structured transformation, start in `data-processing/` before adding multi-agent topology or MCP layering.

### Cross-platform before framework hype
If the task spans platforms, start in `cross-platform-development/` before letting stack preference dominate.
