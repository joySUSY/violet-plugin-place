# Ensemble Team Governance

## Purpose

Define the canonical governance model for ensemble or multi-agent teams inside `developer-tool`.

This document distills the strongest reusable lessons from team-formation donors:

- team size should match project reality
- role selection should be evidence-based
- coordination artifacts must be explicit
- conventions should be formed, not assumed

## Core Rule

A multi-agent team is not just a list of experts.
It is a governed working system.

A good team system defines:

- who exists
- why each role exists
- how decisions are made
- where conventions are recorded
- how disagreement resolves
- how retrospection changes the system over time

---

## Pattern 1 — Tiered Team Presets

A healthy ensemble system offers team presets by scope.

Typical posture:

- **solo-plus** for narrow or exploratory work
- **lean** for most real projects
- **full** for large or highly cross-functional projects

This prevents a common anti-pattern:

- overbuilding team overhead for small projects

A team model should scale up only when governance cost is justified.

---

## Pattern 2 — Research Roles, Don't Memorize Them

Strong ensemble governance does not rely on a frozen list of famous expert names.

Instead, it asks:

- what domain is actually needed?
- what authority or perspective is genuinely relevant here?
- what complementary heuristics does the team need?

This is a powerful principle for `developer-tool`:

> team composition should be context-shaped, not cargo-culted.

---

## Pattern 3 — Profiles Need Compression and Boundaries

Full profiles are useful, but runtime systems need compressed forms.

That means:

- rich role/profile docs for deep understanding
- compressed active-context forms for practical execution

This same principle appears in plugin-first shell design:

- doctrine can be rich
- runtime activation must stay selective and compressed

---

## Pattern 4 — Coordinator Is a Runtime Surface

A real ensemble needs an explicit coordinator role.

The coordinator should own:

- topic sequencing
- discussion order
- role activation
- decision recording
- handoff discipline

The coordinator should **not** own:

- all domain reasoning directly
- silent substitution for missing team discussion

This is important because it mirrors heavy-engine shell design:

- there must be a clear orchestration surface
- but orchestration must not swallow doctrine or roles whole

---

## Pattern 5 — Working Conventions Must Be Deliberate

Good team systems do not merely generate profiles.
They generate conventions and governance artifacts.

Examples:

- team structure in `AGENTS.md`
- project owner constraints in `PROJECT.md`
- ADR or architecture decisions in architecture docs
- glossary and deferred-items docs

The deep principle is:

> team behavior becomes durable only when written into canonical surfaces.

That is directly aligned with our current canonical deep-fusion effort.

---

## Pattern 6 — Formation Session Before Heavy Execution

Before a team starts operating at scale, it should resolve:

- decision style
- definition of done
- conflict resolution
- review expectations
- retrospective cadence
- architecture principles

This formation session is the social equivalent of a plugin shell blueprint.

Without it, runtime coordination looks active, but is structurally weak.

---

## Pattern 7 — Retrospectives Are Event-Driven Learning Loops

The best team donors treat retrospectives as:

- event-driven
- role-authored
- evidence-based
- action-item-producing

This matters because `developer-tool` also needs runtime memory and workflow doctrine that learns from repeated patterns instead of just storing transcripts.
