# Memory Creation and Shaping Cluster

**Authors: Joysusy & Violet Klaudia**

> 先设计记忆的形状，再要求运行时去记住它。
> Shape memory deliberately before asking the runtime to remember it.

## 目的

这份 staged cluster 文档解释 `v3-expansion/` 中 **memory creation / shaping** 这一组的角色。

只有在 canonical memory references 已经完成问题定向之后，而剩余问题真正变成以下之一时，才应该进入这一组：
- 创建 memory-oriented skills 或 memory artifacts
- 塑造 memory structure 或 retrieval-learning workflow
- 探索尚未适合直接 flatten 进 canonical doctrine 的 meta-memory 技术

## 本集群包含的模块

- `advanced-memory-skill-creator/`
- `agent-memory-skills/`
- `memory-systems/`
- `memory-retrieval-learning/`

## 什么时候打开这一组

只有当问题已经不再是：
- “我应该先 recall 什么？”
- “我应该走哪条 history lane？”
- “continuity 是怎么工作的？”

而转变成：
- “memory-oriented skill / artifact 应该怎么构造？”
- “memory system 或 retrieval-learning 资产应该被塑造成什么样？”
- “除了 canonical refs 之外，这里还有哪些更偏 creation / shaping 的已适配材料？”

这时才进入这一组。

## 这一组不是什么

这一组不是：
- recall 的第一线 doctrine
- continuity control plane
- memory architecture 的默认 source of truth

这些问题应先回到：
- `../../references/ai-agent-memory/recall-before-act.md`
- `../../references/ai-agent-memory/memory-systems.md`
- `../../references/ai-agent-memory/continuity-control-plane.md`

## 未来更可能的收束方向

随着后续 deep fusion 推进，这一组更可能产出：
- 更多 memory-creation guidance 被提升进 canonical refs
- 更系统的 evaluation / methodology support doctrine
- 更清楚地区分 artifact creator 与 methodology 模块

在那之前，它仍应被视为 shaped staging pool，而不是 canonical lane。
