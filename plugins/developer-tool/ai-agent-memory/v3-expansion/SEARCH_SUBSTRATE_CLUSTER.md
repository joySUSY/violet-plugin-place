# Search Substrate Cluster

> Heavy search belongs behind doctrine, not in front of it.
> 重型搜索基底必须服从 doctrine，而不是站到 doctrine 前面。

## Purpose

This staged cluster doc explains the **search substrate** side of `v3-expansion/`.

At the moment, this cluster is effectively centered on one heavy operational asset:

- `coding_agent_session_search-main/`

It exists because some retrieval problems are too large, too fragmented, or too cross-source to be handled by lighter lane utilities alone.

## Module in This Cluster

- `coding_agent_session_search-main/`

## When to Open This Cluster

Open this cluster only when the canonical retrieval doctrine has already determined that:

- lighter session/history lanes are insufficient
- indexed multi-source recall is needed
- high-volume or cross-machine retrieval is the real problem

This cluster is appropriate for:

- cross-source retrieval
- high-volume indexed search
- semantic/hybrid recall substrate work
- heavy retrieval infrastructure questions

## What This Cluster Is NOT

This cluster is not:

- the first line of recall doctrine
- a substitute for history-retrieval lane selection
- a justification to skip canonical memory references

If the question is still about choosing the retrieval lane, go back to:

- `../../references/ai-agent-memory/history-retrieval-patterns.md`

## Likely Future Convergence

This cluster is unlikely to be fully flattened into doctrine.

The more realistic future is:

- doctrine continues to absorb lane-selection logic
- the heavy search substrate remains an operational/runtime asset
- wrappers, audits, or narrower support notes may be promoted upward

In other words, this is the clearest example in `v3-expansion/` where **retention as substrate** is more honest than forced flattening.
