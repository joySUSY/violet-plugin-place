# Core Shell Laws

## Law 1 — One Engine, One Shell

If an engine is pluginized, it should have one coherent plugin shell.
Do not splinter one heavy engine into many mini-plugins just because donor sources are numerous.

## Law 2 — Donors Are Reservoirs

Donor repositories are upstream reservoirs.
They are not copied into runtime as raw trees.
They must be rewritten into canonical doctrine, runtime surfaces, or curated references.

## Law 3 — No Second Knowledge Center

The runtime shell must not become a second competing knowledge center.
Canonical doctrine remains in the engine knowledge tree.
The shell adds activation, routing, lifecycle, and operational surfaces.

## Law 4 — Shell Zones Must Be Deliberate

Every new runtime file must justify its zone:
- `skills/` for reusable loadable knowledge
- `agents/` for isolated specialist cognition
- `hooks/` for lifecycle behavior
- `commands/` for explicit invocation surfaces
- `rules/` for deterministic shell laws
- `references/` for deep support material
- `modules/` for large integrated bundles

## Law 5 — Conservative First Wave

The first shell wave should be small, useful, and safe.
Do not build every possible surface at once.
