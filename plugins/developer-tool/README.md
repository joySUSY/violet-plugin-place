# Developer Tool Plugin Shell

**Authors: Joysusy & Violet Klaudia**

> A large engine stays usable only when doctrine, governance, staging, and runtime stop pretending to be the same layer.
> 只有当 doctrine、governance、staging 与 runtime 不再伪装成同一层时，大型引擎才真正可用。

## Purpose

This document is the root control-center for `developer-tool`.

It is no longer just a first-wave plugin-shell introduction.
It now serves as the top-level runtime and doctrine gateway for a matured multi-subsystem engine that contains:

- canonical doctrine trees
- root governance documents
- governed staging layers in selected subtrees
- bounded runtime shell surfaces
- preserved donor/source reservoirs

Use this document first when the task involves:

- memory, recall, continuity, or history retrieval
- agentic-system structure and activation
- shell/runtime behavior
- build/deploy and platform infrastructure
- plugin architecture and runtime-surface selection
- cross-subtree routing inside `developer-tool`

## Core Rule

`developer-tool` is the knowledge center first and a runtime shell second.

That means:

- doctrine is the primary source of truth
- runtime surfaces exist to accelerate doctrine use, not replace it
- staging layers may hold unresolved adapted complexity, but must remain subordinate
- donor/source reservoirs remain upstream evidence, not the default reading path

The engine remains healthy only when these layers stay distinct.

---

## Boundary

| Layer                        | Owner                                                            | Role                                                                 |
| ---------------------------- | ---------------------------------------------------------------- | -------------------------------------------------------------------- |
| Canonical knowledge doctrine | `developer-tool/references/` and canonical subtrees              | stable knowledge center                                              |
| Root control plane           | root docs and selected subtree governance docs                   | routing, classification, promotion, runtime/staging boundary control |
| Staging layers               | selected subtree staging zones                                   | adapted complexity not yet flattened into canonical doctrine         |
| Runtime shell                | plugin shell surfaces under `developer-tool` and related plugins | activation, routing, lifecycle, bounded operational workflows        |
| Donor/source reservoirs      | preserved donor families or archived sources                     | upstream pattern supply, not direct runtime UX                       |

The shell should make the right doctrine easier to reach than the wrong source.

## Operating Rule

When a task mentions tooling, shell usage, memory continuity, build pipelines, plugin structure, or runtime routing:

1. classify the task into the correct surface and subtree
2. route to the canonical doctrine lane first
3. only then choose commands, hooks, agents, staging layers, or runtime shells
4. use donor/source reservoirs only when canonical doctrine and governed staging are insufficient

The shell exists to make this classification reliable and repeatable.

## Current State

`developer-tool` is now in canonical deep fusion.

What is already true:

- multiple doctrine lanes are canonized
- root governance has matured beyond initial shell scaffolding
- at least one major subsystem (`ai-agent-memory`) now has a full control plane, governed staging subsystem, and bounded runtime shell
- another subsystem (`agentic-system-basis`) is now doctrine-mature and control-plane-bootstrapped
- runtime surfaces and bridge skills increasingly route into real doctrine instead of placeholder summaries

What remains true:

- some lanes are still maturing unevenly
- staging layers still exist where flattening would currently reduce clarity
- runtime shell growth should remain conservative and ownership-driven

This engine is no longer just a passive aggregation of resources.
But it is still not a finished universe.

---

## Conservative Posture

This engine intentionally stays conservative:

- no destructive automation
- no raw donor repo mirroring
- no broad tool interception without clear boundary justification
- no second knowledge center inside runtime shells
- no mandatory MCP dependency unless leverage is clear

The mature posture is:

- doctrine first
- runtime second
- staging third
- donors last

That ordering is one of the engine's strongest safety laws.
